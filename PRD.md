# AirPool — 私有机场平台 需求文档

> 构建一个简洁、健壮、快速的私有代理（"机场"）平台。
> 面向拥有多台海外服务器的个人或小团队，解决节点管理、智能调度、抗封锁、多平台接入的完整链路。

---

## 一、背景与愿景

### 1.1 痛点

| # | 痛点 | 影响 |
|---|------|------|
| 1 | 单一协议（WireGuard）被 GFW 精准识别，端口频繁被封 | 每次被封需手动重装、换端口、重新配置所有客户端 |
| 2 | 多台海外服务器各自独立，缺乏统一管理 | 不知道哪台快、哪台慢 |
| 3 | 节点增减需要逐台配置、逐设备同步 | 运维成本随节点和设备数量指数增长 |
| 4 | 节点故障或被限速时，客户端无法自动感知 | 用户体验断崖式下降，需手动切换 |
| 5 | 5 种操作系统的设备需要统一接入 | 配置碎片化，维护困难 |

### 1.2 愿景

**一句话**：像商业机场一样好用，但完全自主可控。

- 新服务器一条命令上线，自动加入调度
- 服务器下线、故障、限速，客户端无感自动切换
- 五个平台一个订阅链接，开箱即用
- 管理员通过 Web 面板掌控全局

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| **简洁优先** | 能不做的就不做，能少一个组件就少一个组件 |
| **信任客户端** | 客户端（Mihomo）已有成熟的测速和切换机制，不重复造轮子 |
| **信任服务商** | 流量统计、限速、停机由服务商处理，系统只需感知结果 |
| **故障自愈** | 所有异常场景都应自动恢复，不依赖人工干预 |
| **最小侵入** | 节点侧只装一个代理服务（Hysteria2），不装任何额外 agent |

```
                        ┌─────────────────────────────┐
                        │     控制面板（Admin Panel）    │  [P1]
                        │     节点管理 / 池管理 / 日志   │
                        └──────────┬──────────────────┘
                                   │ API
                        ┌──────────▼──────────────────┐
                        │     配置中心（API Server）     │
                        │                              │
                        │  • 节点注册/注销              │  推荐独立小 VPS
                        │  • 主动健康检查（拉模型）      │  （$3-5/月）
                        │  • 订阅配置动态生成            │
                        │  • 告警推送                   │  Go 单二进制 + SQLite
                        │                              │
                        │  每 60 秒主动检查所有节点 ──┐  │
                        └───┬────────────────────────┘│─┘
                            │                        │
              ┌─────────────▼──┐     ┌───────────────▼────────┐
              │  订阅 HTTP 接口 │     │    代理节点集群          │
              │ (客户端通过代理  │     │                        │
              │  直接访问)      │     │  ┌── 🇺🇸 US 池 ────────┐│
              └──────┬─────────┘     │  │ us-a3f2  ✓  32ms   ││
                     │               │  │ us-b7e1  ✓  45ms   ││
        ┌────────────▼────────────┐  │  └────────────────────┘│
        │    客户端（全平台）       │  │  ┌── 🇯🇵 JP 池 ────────┐│
        │                        │  │  │ jp-c4d9  ✓  68ms   ││
        │  iOS      — Stash      │  │  └────────────────────┘│
        │  Android  — CMFA       │  │  ┌── 💎 Premium 池 ───┐│
        │  macOS    — Clash Verge│  │  │ pr-e2a1  ✓  40ms   ││
        │  Windows  — Clash Verge│  │  └────────────────────┘│
        │  Linux    — mihomo CLI │  └────────────────────────┘
        │                        │
        │  统一订阅 URL           │  节点不需要知道配置中心在哪
        │  url-test 自动选最快    │  配置中心迁移 = 零节点操作
        └────────────────────────┘
```

**关键决策**：

| 决策 | 说明 |
|------|------|
| 拉模型健康检查 | 配置中心主动检查节点，节点不需要知道配置中心地址，迁移零成本 |
| 节点零依赖 | 节点上只有 Hysteria2，没有心跳脚本、没有 agent、没有配置中心地址 |
| 配置中心推荐独立部署 | 便宜小 VPS（$3-5/月），和代理节点分离 |
| 不用 CDN，不用域名 | 客户端通过代理直接访问配置中心 HTTP 接口 |
| P0 使用 HTTP | 通信链路经过 Hysteria2 已加密，不需要 HTTPS |

**已知限制**（可接受）：

| 场景 | 影响 | 接受原因 |
|------|------|---------|
| 配置中心挂了 | 新节点无法注册，健康检查暂停，管理员无法操作 | 已接入用户不受影响（本地缓存），重启即恢复 |
| 全部节点宕机 | Telegram 告警在国内收不到 | 极端场景，邮件告警作为备用 |

## 三、功能需求

### M1 — 节点生命周期管理 `[P0]`

#### F1.1 一键部署

| 项目 | 要求 |
|------|------|
| 部署方式 | 新服务器执行一条命令，完成全部安装、配置、注册 |
| 必填参数 | `--server` 配置中心地址 / `--token` API Token / `--pool` 所属池 |
| 可选参数 | `--name` 自定义名称（不传则自动生成：`{pool}-{随机4位}`，如 `us-a3f2`） |
| 部署示例 | `curl -sL http://配置中心:8080/install.sh \| bash -s -- --server 1.2.3.4:8080 --token ak_xxx --pool us` |
| 自动执行 | 从配置中心获取 Hysteria2 密码 → 安装 Hysteria2 → 生成证书 → 配置端口跳跃 → 启动服务 → 向配置中心注册 |
| 安全 | install.sh 不含敏感信息，Token 通过参数传入 |
| 密码管理 | Hysteria2 密码由配置中心统一管理，部署时自动写入节点配置 |
| 幂等性 | 重复执行不会破坏已有配置，可用于修复 |
| 注册后 | 节点不再需要与配置中心通信，由配置中心主动检查节点健康状态 |

#### F1.2 节点注册与健康检查

| 项目 | 要求 |
|------|------|
| 注册 | 部署脚本向配置中心 `POST /api/nodes` 注册（name、pool、ip），一次性操作 |
| 健康检查 | 配置中心每 60 秒主动 TCP 连接节点的 443 端口（Hysteria2 masquerade） |
| 判定逻辑 | 连通 = online，连续 3 次不通（3 分钟）= offline |
| 自动上线 | 注册成功后，下一次健康检查通过即标记 online，加入订阅配置 |
| 自动下线 | 健康检查连续失败 3 次，标记 offline，从订阅配置移除，发送告警 |
| 自动恢复 | 节点恢复后健康检查自动通过，重新标记 online，加回订阅配置 |
| 手动管理 | 管理员可通过 API 手动上线、下线、迁移池 |

#### F1.3 一键升级

| 项目 | 要求 |
|------|------|
| 升级方式 | 节点上执行 `airpool-upgrade` 一条命令 |
| 下载来源 | 从固定 URL（如 GitHub Releases）下载新版本，不依赖配置中心 |
| 配置保留 | 只替换 Hysteria2 二进制，保留节点配置、证书、端口跳跃规则 |
| 配置迁移 | 新版本若有配置格式变更，升级脚本自动完成迁移 |
| 回滚 | 升级前自动备份，失败自动回滚并重启 |

#### F1.4 节点退出

| 项目 | 要求 |
|------|------|
| 主动退出 | 执行 `airpool-uninstall` 一键卸载；管理员通过 API 删除节点记录 |
| 被动退出 | 健康检查持续失败，自动标记 offline |
| 客户端无感 | offline 节点从订阅移除，客户端下次拉取自动生效 |

### M2 — 池与调度 `[P0]`

#### F2.1 池管理

| 项目 | 要求 |
|------|------|
| 池定义 | 管理员自由创建池，每个池有名称和描述 |
| 分类方式 | 按地域（us / jp / hk）、按价格（premium / standard / backup）或任意维度 |
| 节点归属 | 一个节点只属于一个池 |
| auto 池 | 系统内置 `auto` 池，自动包含所有在线节点，始终存在，不可删除 |
| 管理操作 | 创建 / 编辑 / 删除池（通过 API，P1 后通过面板） |

#### F2.2 智能调度

| 项目 | 要求 |
|------|------|
| 调度策略 | **统一使用 `url-test`**：池内自动选延迟最低的节点，谁快用谁 |
| 延迟检测 | 客户端每 30 秒对池内所有节点测速 |
| 超时阈值 | 2000ms 内无响应视为不可用 |
| 切换容差 | tolerance 50ms，避免在延迟接近的节点间频繁跳转 |
| 故障切换 | 当前节点不可达时，自动切到池内次快节点（< 30 秒） |
| 全程无感 | 所有调度由客户端自动完成，用户无需任何操作 |

> **设计哲学**：
>
> **不做权重，不做服务端限速，不做流量统计。**
>
> - 谁快用谁（url-test），不需要人为指定优先级
> - 节点慢了（被限速/带宽满），延迟自然升高，客户端自动切走
> - 节点挂了（流量耗尽停机），健康检查失败，自动从订阅移除
> - 节点恢复了（服务商重置/机器重启），健康检查恢复，自动重新加入
> - 节点 IP 被封，健康检查正常（海外到海外），但客户端侧不可达，url-test 自动跳过
>
> **系统只负责维护"哪些节点在线"这一个事实，选择交给客户端。**

### M3 — 抗封锁 `[P0]`

#### F3.1 协议

| 项目 | 要求 |
|------|------|
| 主协议 | Hysteria2（基于 QUIC，伪装 HTTP/3，速度最快） |
| 备选协议 | VLESS + Reality（伪装 TLS 1.3，UDP 被限时的降级方案）`[P2]` |

#### F3.2 端口跳跃

| 项目 | 要求 |
|------|------|
| 实现方式 | iptables 将 20000-40000（2 万端口）转发到主端口 443 |
| 客户端配置 | `server: IP:443,20000-40000`，客户端自动在范围内跳转 |
| 效果 | 封几个端口完全无感 |

#### F3.3 伪装

| 项目 | 要求 |
|------|------|
| Hysteria2 | masquerade 反代真实网站，主动探测返回正常网页 |

### M6 — 配置中心 `[P0]`

#### F6.1 部署

| 项目 | 要求 |
|------|------|
| 形态 | 单 Go 二进制 + SQLite 文件 |
| 推荐部署 | 独立小 VPS（$3-5/月），长期稳定 |
| 备选部署 | 跑在某台代理节点上（省钱但耦合风险） |
| 资源占用 | 内存 < 50MB |
| 启动 | `./airpool-server`，systemd 管理 |
| 初始化 | 首次启动自动创建数据库、生成 API Token、Subscribe Token、Hysteria2 密码 |

#### F6.2 配置项

| 配置项 | 说明 |
|--------|------|
| `listen` | 监听地址和端口（默认 `:8080`） |
| `api_token` | 节点注册认证 + 管理 API 认证（首次自动生成） |
| `subscribe_token` | 客户端订阅认证（首次自动生成） |
| `hysteria2_password` | Hysteria2 全局密码（所有节点共用） |
| `check_interval` | 健康检查间隔（默认 60 秒） |
| `check_timeout` | 单次检查超时（默认 5 秒） |
| `offline_threshold` | 连续失败多少次标记 offline（默认 3 次） |
| `telegram_bot_token` | Telegram 告警 Bot Token（可选） |
| `telegram_chat_id` | 告警接收人 Chat ID（可选） |
| `alert_email` | 备用邮件告警地址（可选） |

#### F6.3 告警 `[P0]`

| 项目 | 要求 |
|------|------|
| 触发条件 | 节点从 online → offline |
| 恢复通知 | 节点从 offline → online |
| 主通道 | Telegram Bot |
| 备用通道 | 邮件（国内可直接收，不依赖代理） |

#### F6.4 迁移

| 项目 | 要求 |
|------|------|
| 迁移方式 | 复制二进制 + SQLite 文件到新机器，启动即可 |
| 节点侧 | **零操作**——节点不知道也不需要知道配置中心在哪，配置中心从新 IP 继续检查所有节点 |
| 客户端侧 | 管理员发新订阅 URL 给用户更新（迁移期间旧缓存仍可用，不影响上网） |
| 备份 | 复制 SQLite 文件即可 |
| 升级 | 替换二进制，重启 |

#### F6.5 P0 管理方式

P0 无面板，管理员通过 curl 调 API：

```bash
API=http://配置中心IP:8080
TOKEN=ak_xxxxxxxx

# 创建池
curl -X POST $API/api/pools -H "Authorization: $TOKEN" \
  -d '{"name": "us", "description": "美国节点"}'

# 查看所有节点
curl $API/api/nodes -H "Authorization: $TOKEN"

# 下线一个节点
curl -X PUT $API/api/nodes/us-a3f2/offline -H "Authorization: $TOKEN"

# 迁移节点到另一个池
curl -X PUT $API/api/nodes/us-a3f2/pool -H "Authorization: $TOKEN" \
  -d '{"pool": "premium"}'

# 删除节点
curl -X DELETE $API/api/nodes/us-a3f2 -H "Authorization: $TOKEN"

# 获取订阅 URL
echo "$API/api/subscribe?token=st_xxxxxxxx"
```

## 四、技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 代理协议 | Hysteria2 | 最快、轻量、端口跳跃、伪装 HTTP/3 |
| 客户端内核 | Mihomo (Clash Meta) | 跨平台生态最成熟、url-test 自动调度 |
| 配置中心 | Go 单二进制 | 轻量、无依赖、交叉编译方便 |
| 数据存储 | SQLite | 单文件、零运维 |
| 控制面板 | 内嵌 Web 前端（Go embed）`[P1]` | 不额外增加组件 |
| 健康检查 | 配置中心内置（TCP connect to 443） | 利用 Hysteria2 masquerade 端口，节点零配置 |
| 告警 | Telegram Bot API + 邮件 | 互补：Telegram 即时，邮件兜底 |

**原则**：
- 节点上只有一个运行组件：Hysteria2
- 配置中心只有一个运行组件：Go 二进制 + SQLite
- 没有心跳脚本、没有 agent、没有 Redis、没有消息队列、没有 CDN、没有域名依赖

## 五、核心流程

### 5.1 配置中心初始化（一次性）

```
管理员准备一台海外 VPS（推荐 $3-5/月小机器）
       │
       ▼
下载 airpool-server 二进制，启动
       │
       ▼
首次启动自动：
  ├─ 创建 SQLite 数据库
  ├─ 生成 API Token（节点注册 + 管理用）
  ├─ 生成 Subscribe Token（客户端订阅用）
  ├─ 生成 Hysteria2 全局密码
  ├─ 启动健康检查调度器（每 60 秒）
  └─ 输出到终端：
       API Token:       ak_xxxxxxxx
       Subscribe Token: st_xxxxxxxx
       Hy2 Password:    hp_xxxxxxxx
       订阅 URL:        http://IP:8080/api/subscribe?token=st_xxxxxxxx
```

### 5.2 新节点上线

```
管理员先创建池
  curl -X POST http://IP:8080/api/pools -H "Authorization: ak_xxx" \
    -d '{"name": "us"}'
       │
       ▼
在新服务器上执行部署命令
  curl -sL http://IP:8080/install.sh | bash -s -- \
    --server IP:8080 --token ak_xxx --pool us
       │
       ▼
脚本自动执行（约 30 秒）
       │
       ├─ 从配置中心获取 Hysteria2 密码
       ├─ 安装 Hysteria2
       ├─ 生成自签证书
       ├─ 写入 Hysteria2 配置（密码 + masquerade + 端口 443）
       ├─ 配置 iptables 端口跳跃 (20000-40000 → 443)
       ├─ 启动 Hysteria2
       └─ 向配置中心注册 POST /api/nodes {name: us-a3f2, pool: us, ip: x.x.x.x}
              │
              ▼
       注册完成，部署脚本退出
       节点不再需要与配置中心有任何通信
              │
              ▼
       配置中心下一轮健康检查（≤ 60 秒）检测到节点在线
       自动加入 us 池 + auto 池的订阅配置
              │
              ▼
       客户端下次拉取即获得新节点（≤ 1 小时自动 / 手动刷新立即）
```

### 5.3 节点故障自动处理

```
故障发生
   │
   ├─ 场景A：节点变慢（服务商限速 / 网络波动 / 带宽满）
   │    → 延迟升高
   │    → 客户端 url-test 30 秒内检测到
   │    → 自动切到池内延迟更低的节点
   │    → 节点恢复后延迟降回，客户端自动切回
   │
   ├─ 场景B：节点完全不可达（宕机 / 服务商停机 / 流量耗尽停机）
   │    → 客户端测速超时，立即切到其他节点
   │    → 配置中心健康检查连续 3 次失败（3 分钟），标记 offline
   │    → 从订阅配置移除 + 发送告警
   │    → 节点恢复 → 健康检查自动通过 → 重新加入 + 发送恢复通知
   │
   ├─ 场景C：端口被封
   │    → 2 万端口跳跃范围，客户端自动跳到未封端口
   │    → 用户完全无感
   │
   └─ 场景D：IP 被封
        → 客户端 url-test 超时，自动切到其他节点（< 30 秒）
        → 健康检查正常（配置中心海外→节点海外，不过墙）
        → 节点留在订阅里但客户端自动跳过，不影响体验
        → 管理员可手动下线或更换 IP

全程无需人工干预（IP 被封除外，需管理员后续处理）
```

### 5.4 节点升级

```
管理员 SSH 到节点，执行 airpool-upgrade
       │
       ├─ 备份当前 Hysteria2 二进制和配置
       ├─ 从 GitHub Releases 下载最新版本
       ├─ 停止 Hysteria2（客户端自动切到其他节点）
       ├─ 替换二进制，迁移配置（如有变更）
       ├─ 启动 Hysteria2
       ├─ 自检（验证 443 端口监听）
       │    ├─ 通过 → 完成
       │    └─ 失败 → 自动回滚并重启
       └─ 停服 < 5 秒，用户无感
```

## 六、非功能需求

| 维度 | 要求 |
|------|------|
| **部署效率** | 新节点从裸机到可用 < 1 分钟 |
| **故障切换** | < 30 秒（客户端 url-test 自动完成） |
| **配置同步** | ≤ 1 小时自动 / 手动刷新立即 |
| **抗封锁** | 2 万端口跳跃 + 协议伪装 |
| **安全** | 三个独立 Token：API Token + Subscribe Token + Hysteria2 密码 |
| **轻量** | 节点侧：只有 Hysteria2，内存 < 20MB |
| **轻量** | 配置中心：单 Go 二进制 + SQLite，内存 < 50MB |
| **跨平台** | iOS / Android / macOS / Windows / Linux |
| **容灾** | 配置中心挂了不影响已接入用户（客户端本地缓存） |
| **可迁移** | 配置中心迁移 = scp 一个文件 + 启动，节点零操作 |
